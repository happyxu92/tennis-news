# 网球赛事信息采集与微信公众号发布系统 PRD

## 1. 项目概述

### 1.1 项目名称
网球赛事信息采集与微信公众号自动发布系统

### 1.2 项目背景
需要使用 Python 开发一个自动化系统，持续采集全球主要网球赛事的赛程信息和比赛结果信息，并将符合规则的内容自动发布为微信公众号图文文章。

系统需要覆盖四大满贯、全部 ATP 赛事、全部 WTA 赛事，并具备稳定的数据同步、内容生成、去重判定、按规则发布的能力。

### 1.3 项目目标
1. 自动采集四大满贯、ATP、WTA 赛事信息。
2. 自动同步每场比赛的最新赛程安排和比赛结果。
3. 将不同赛事的赛程内容分开发布。
4. 仅对重点比赛发布单场比赛结果文章。
5. 当没有新增赛程或新增结果时，不发布任何文章。
6. 构建清晰、可维护的 Python 项目架构。
7. 拆分独立模块，至少包括“数据采集模块”和“微信公众号发布模块”。

---

## 2. 产品目标

### 2.1 核心目标
构建一个稳定的自动化内容生产系统，实现“数据采集 -> 数据比对 -> 内容生成 -> 微信发布”的全链路闭环。

### 2.2 业务目标
1. 覆盖高价值网球赛事信息。
2. 保证赛程和赛果信息尽可能及时。
3. 减少人工整理赛事资讯的成本。
4. 避免重复发布和无变化发布。
5. 为后续扩展更多数据规则和内容模板留出空间。

### 2.3 成功标准
1. 能稳定获取指定赛事数据。
2. 能识别赛程变化和新增结果。
3. 能按规则自动生成公众号文章。
4. 能成功调用微信公众号接口完成图文发布。
5. 能记录发布历史，避免重复发文。

## 3. 用户与使用场景

### 3.1 用户角色
1. 系统管理员/运营者
2. 微信公众号内容运营者

### 3.2 主要使用场景
1. 每天自动抓取所有目标赛事赛程。
2. 当某赛事赛程发生变化时，发布对应赛事的赛程更新文章。
3. 当重点比赛结束后，自动发布单场赛果文章。
4. 当当天没有新变化时，系统不发文。
5. 运营者可查看同步结果、发布状态、失败原因。

## 4. 需求范围

### 4.1 数据覆盖范围
1. 四大满贯赛事
2. 所有 ATP 赛事
3. 所有 WTA 赛事

### 4.2 内容范围
1. 每场比赛最新赛程安排
2. 每场比赛最新比赛结果

### 4.3 发布范围
1. 赛事赛程文章按赛事分开发布
2. 比赛结果文章按单场比赛单独发布
3. 只发布重点比赛结果，不发布全部比赛结果

### 4.4 重点比赛定义
重点比赛满足以下任一条件即可：
1. 种子选手比赛
2. 国内选手比赛
3. 后续可扩展为配置规则，例如明星选手、决赛、半决赛、焦点对决

## 6. 业务规则

### 6.1 赛事赛程发布规则
1. 一篇文章对应一个赛事在某个时间点的赛程更新。
2. 不同赛事必须分开发布。
3. 仅当该赛事存在新增比赛、时间变化、场地变化、轮次变化、对阵变化、状态变化时才发布。
4. 如果抓取结果与上一次内容一致，则不发布。

### 6.2 比赛结果发布规则
1. 每场重点比赛结果单独发布一篇文章。
2. 仅在比赛首次明确完赛且结果完整时发布。
3. 若非重点比赛，即使完赛也不发布。
4. 若比赛结果没有变化，则不重复发布。
5. 若赛后比分被官方修正，首版默认不自动再次发布更正文章，仅记录修正信息。
6. 后续可扩展为“比分修正后追加更正文章”的可配置能力。

### 6.3 无变化不发布规则
1. 没有新增赛程变更，不发布赛事赛程文章。
2. 没有新增重点比赛结果，不发布赛果文章。
3. 已发布过相同内容，不重复发文。

### 6.4 发布时间策略
1. 赛程信息定时抓取并比对。
2. 比赛结果高频抓取并比对。
3. 文章发布采用异步任务方式执行。
4. 微信接口发布结果需要异步轮询确认。

## 7. 功能需求

## 7.1 数据采集模块

### 7.1.1 赛事同步
1. 获取目标赛事列表。
2. 同步赛事基础信息，包括赛事名称、级别、地点、场地类型、起止日期等。
3. 区分赛事所属巡回赛类别，例如 Grand Slam、ATP、WTA。

### 7.1.2 赛程同步
1. 获取赛事下所有比赛赛程。
2. 获取比赛轮次、开赛时间、场地、选手、状态等信息。
3. 支持识别比赛延期、取消、改时间、改场地等变化。

### 7.1.3 赛果同步
1. 获取比赛最终比分、胜者、比赛状态。
2. 支持识别比赛从 scheduled/live 进入 finished。
3. 支持识别比赛结果字段更新。

### 7.1.4 数据标准化
1. 统一不同数据源的字段结构。
2. 统一赛事、球员、比赛、时间、状态等核心对象格式。
3. 统一时间存储标准，内部使用 UTC，展示时转换为北京时间。

---

## 7.2 变化检测模块

### 7.2.1 赛程变化检测
1. 对比当前抓取结果与上次快照。
2. 识别新增比赛。
3. 识别比赛时间变化。
4. 识别场地变化。
5. 识别轮次变化。
6. 识别对阵变化。
7. 识别状态变化。

### 7.2.2 赛果变化检测
1. 识别比赛首次完赛。
2. 识别比分新增。
3. 识别胜者新增。
4. 识别关键结果字段变化。
5. 根据重点比赛规则决定是否生成发布任务。

### 7.2.3 去重与幂等
1. 相同业务内容不重复创建发布任务。
2. 已成功发布的相同内容不重复发送到微信。
3. 失败任务允许重试，但不能生成重复文章。

---

## 7.3 内容生成模块

### 7.3.1 赛事赛程文章
1. 生成赛事专属标题。
2. 生成赛事赛程正文。
3. 展示更新时间。
4. 展示比赛轮次、对阵、时间、场地、状态。
5. 可突出显示重点比赛。

### 7.3.2 单场赛果文章
1. 生成单场比赛结果标题。
2. 展示赛事名称、轮次、双方选手、比分、胜者、比赛时间。
3. 对重点比赛给出简短摘要。
4. 标题和摘要需适合微信公众号图文形式。

### 7.3.3 内容模板
1. 支持赛程文章模板。
2. 支持赛果文章模板。
3. 后续支持不同赛事级别使用不同模板风格。
4. 模板内容支持按规则插入重点标记。

---

## 7.4 微信公众号发布模块

### 7.4.1 微信接口能力
1. 获取 access token。
2. 上传封面图。
3. 上传正文内图片。
4. 创建草稿。
5. 提交发布。
6. 查询发布状态。

### 7.4.2 发布流程
1. 根据发布任务生成图文 HTML。
2. 调用微信草稿接口保存文章。
3. 调用发布接口正式发布。
4. 轮询发布状态直至完成。
5. 记录发布结果、文章 URL、发布时间。

### 7.4.3 发布失败处理
1. 记录失败原因。
2. 支持重试。
3. 支持区分草稿失败、发布失败、审核失败、平台拒绝等状态。

## 8. 非功能需求

### 8.1 稳定性
1. 数据源异常时不能导致全系统崩溃。
2. 单个发布失败不能阻塞后续任务。
3. 系统应具备重试和错误日志能力。

### 8.2 可维护性
1. 模块职责清晰。
2. 数据采集与发布逻辑解耦。
3. 新增数据源时不需要重写核心业务流程。
4. 新增文章模板时不影响采集逻辑。

### 8.3 可扩展性
1. 可扩展更多重点比赛判定规则。
2. 可扩展更多内容模板。
3. 可扩展多数据源策略。
4. 可扩展为多发布渠道。

### 8.4 性能
1. 支持高频同步。
2. 支持一天内较多赛果文章发布任务。
3. 发布任务与抓取任务应异步解耦。

### 8.5 观测性
1. 记录抓取日志。
2. 记录数据变化日志。
3. 记录发布日志。
4. 记录微信接口调用日志。
5. 能查询失败原因和任务状态。

## 9. 技术方案

## 9.1 推荐技术栈
1. Python 3.12
2. `httpx` 作为 HTTP 客户端
3. `pydantic` 作为数据模型校验
4. `SQLAlchemy` 作为 ORM
5. `APScheduler` 作为定时任务调度器
6. `Jinja2` 作为文章模板引擎
7. `pytest` 作为测试框架
8. `ruff` 作为格式化与静态检查工具

## 9.2 推荐数据源策略
1. 首选商业体育数据源，例如 `Sportradar Tennis API`
2. 备选 `SportsDataIO Tennis API`
3. 官方 ATP/WTA/大满贯站点仅作为辅助校验来源
4. 不建议将网页抓取作为主方案

推荐理由：
1. 覆盖范围更完整
2. 稳定性更高
3. 接口结构更适合长期维护
4. 降低网页变动导致失效的风险

## 10. 模块设计

## 10.1 crawler 模块
职责：
1. 对接外部数据源
2. 抓取赛事、赛程、赛果
3. 标准化数据结构
4. 输出统一 DTO 给业务层

要求：
1. 每个数据源采用独立 client 或 adapter
2. 不在该模块内直接处理微信发布
3. 保持与发布模块解耦

## 10.2 publisher 模块
职责：
1. 根据业务数据渲染公众号图文内容
2. 调用微信公众号接口
3. 管理草稿和发布状态
4. 返回可追踪的发布结果

要求：
1. 不承担数据采集职责
2. 不直接决定是否发布，由业务层控制
3. 所有微信调用具备清晰的错误处理

## 10.3 services 模块
职责：
1. 编排抓取流程
2. 编排比对逻辑
3. 识别内容变化
4. 创建发布任务
5. 管理任务重试与幂等

## 12. 关键规则设计

## 12.1 重点比赛识别规则
比赛满足以下任一条件即为重点比赛：
1. 任一选手为种子选手
2. 任一选手为中国选手
3. 后续可扩展为配置名单中的关注选手
4. 后续可扩展为半决赛、决赛等重点轮次

## 12.2 赛程文章幂等规则
建议业务键：
`赛事ID + 日期 + 文章类型`

建议去重策略：
1. 对渲染后的内容计算 `content_hash`
2. 若同一业务键下 `content_hash` 未变化，则不发布
3. 若变化，则创建新发布任务

## 12.3 赛果文章幂等规则
建议业务键：
`比赛ID + 文章类型`

建议去重策略：
1. 比赛首次完赛且为重点比赛时生成任务
2. 对最终比分等关键字段生成 `content_hash`
3. 若已存在相同 `content_hash` 的成功发布记录，则跳过

## 13. 发布流程

## 13.1 赛程发布流程
1. 定时抓取赛事赛程
2. 标准化并落库
3. 比对上次快照
4. 识别赛程变化
5. 按赛事生成文章内容
6. 创建发布任务
7. 调用微信草稿接口
8. 调用微信发布接口
9. 轮询发布结果
10. 成功后记录文章地址和发布时间

## 13.2 赛果发布流程
1. 定时抓取比赛结果
2. 标准化并落库
3. 判断比赛是否已完赛
4. 判断是否属于重点比赛
5. 判断是否已发布相同结果
6. 生成单场赛果文章
7. 创建发布任务
8. 调用微信发布链路
9. 记录发布结果

---

## 14. 调度策略

### 14.1 建议同步频率
1. 赛事赛程同步：每 10 分钟一次
2. 比赛结果同步：每 5 分钟一次
3. 发布任务处理：每 1 分钟一次
4. 微信发布状态查询：每 1 到 2 分钟一次

### 14.2 频率设计原则
1. 赛果更新优先级高于赛程更新
2. 抓取与发布解耦
3. 高峰期允许适当调高抓取频率
4. 低峰期可降低频率以减少请求压力

---

## 15. 微信公众号约束与前提

### 15.1 已确认前提
1. 公众号类型为订阅号
2. 账号已认证
3. 发布内容为图文文章
4. 不同赛事文章分开发布
5. 重点比赛结果单独发文
6. 无新增内容不发文

### 15.2 微信接口约束
1. 图文发布为异步流程
2. 草稿与发布接口有调用频率限制
3. 单日发布量需要关注官方配额
4. 发布成功后需要保存文章 URL
5. 图片资源需要走微信素材能力

### 15.3 设计影响
1. 系统需记录完整发布任务状态
2. 系统需控制重试次数
3. 系统需避免高峰日过度发文
4. 系统需具备失败补偿机制

## 16. 风险与应对

## 16.1 数据源风险
风险：
1. 第三方接口收费
2. 覆盖范围或更新频率与预期不符
3. 字段结构后续变更

应对：
1. 优先选稳定商业数据源
2. 在接入前验证重点字段
3. 对 client/adapters 做隔离封装

## 16.2 微信发布风险
风险：
1. 发布接口配额不足
2. 发布异步失败
3. 内容审核或平台限制

应对：
1. 实现发布任务队列
2. 实现状态轮询和失败重试
3. 保留本地内容与任务记录

## 16.3 内容量风险
风险：
1. 大赛日重点比赛数量较多
2. 可能导致单日文章数量偏高

应对：
1. 将重点比赛规则配置化
2. 后续支持按赛事级别或轮次进一步收缩
3. 后续支持合并部分赛果文章策略

## 16.4 时间与时区风险
风险：
1. 数据源时间格式不一致
2. 展示时间可能出现偏差

应对：
1. 系统内部统一存 UTC
